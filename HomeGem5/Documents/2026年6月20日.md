# gem5 SE mode NUMA対応 改造ドキュメント

## 背景

gem5のSE（Syscall Emulation）modeでNUMAシステムのスレッド配置戦略を研究する際、以下の問題が発覚した。

- `GOMP_CPU_AFFINITY`環境変数によるスレッドのCPUピン止めが**完全に無視**されていた
- 物理メモリの割り当てが常に**Node0固定**で、Node1には一切アクセスされなかった

### 根本原因

```
# src/arch/x86/linux/syscall_tbl64.cc（改造前）
{203, "sched_setaffinity", ignoreFunc},  ← 呼ばれても何もしない
```

`sched_setaffinity`（syscall番号203）が`ignoreFunc`に割り当てられており、
libgompが`GOMP_CPU_AFFINITY`を実現するためにこのsyscallを呼んでも完全に無視されていた。

また`seWorkload->allocPhysPages(npages)`の`pool_id`がデフォルト0固定のため、
全メモリがNode0（pool_id=0）から割り当てられていた。

### 改造前の状態（Scatter 4スレッド）

```
system.node0.mem_ctrl.readReqs = 46162
system.node1.mem_ctrl.readReqs = 0
```

## 改造ファイル一覧

### 1. `src/sim/process.hh`

**変更箇所：** `SEWorkload *seWorkload;`の直後に追加

```cpp
// NUMA: thread affinity map (thread_id -> cpu_id)
std::map<int, int> threadAffinityMap;
int coresPerNode = 8;  // Node0: CPU0-7, Node1: CPU8-15

void setThreadAffinity(int tid, int cpu_id) {
    threadAffinityMap[tid] = cpu_id;
}
int getThreadAffinity(int tid) {
    auto it = threadAffinityMap.find(tid);
    return (it != threadAffinityMap.end()) ? it->second : 0;
}
int getNumaNode(int tid) {
    return getThreadAffinity(tid) / coresPerNode;
}
```

**意図：** スレッドIDとCPU番号のマッピングを`Process`オブジェクトに持たせる。
`coresPerNode`はノード構成に合わせて変更する（現在は4P4E×2ノード=8）。

---

### 2. `src/sim/process.cc`

**変更箇所：** `Process::allocateMem()`内、物理ページを割り当てる行

```cpp
// 変更前
const Addr paddr = seWorkload->allocPhysPages(npages);

// 変更後
// NUMA: スレッドのaffinityに基づいてpool_id(=NUMAノード)を決定
int numa_pool_id = 0;
for (auto cid : contextIds) {
    ThreadContext *tc_check = system->threads[cid];
    if (tc_check && tc_check->status() == ThreadContext::Active) {
        numa_pool_id = getNumaNode(tc_check->cpuId());
        break;
    }
}
const Addr paddr = seWorkload->allocPhysPages(npages, numa_pool_id);
```

**意図：** `allocPhysPages`の第2引数`pool_id`にNUMAノード番号を渡す。
`MemPools::populate()`によってノードのAddrRangeごとにpoolが自動生成されるため、
pool_id=0がNode0、pool_id=1がNode1に対応する。

**注意：** `tc->cpuId()`はSMT構成でない限り常に0を返すため、
この方法では`sched_setaffinity`で記録されたaffinityが間接的に参照される。
詳細は「現在の限界」を参照。

### 3. `src/sim/syscall_emul.hh`

**変更箇所：** `schedGetaffinityFunc`の直前に新規関数を追加

```cpp
template <class OS>
SyscallReturn
schedSetaffinityFunc(SyscallDesc *desc, ThreadContext *tc,
                     pid_t pid, typename OS::size_t cpusetsize,
                     VPtr<> cpu_set_mask)
{
#if defined(__linux__)
    if (cpusetsize < sizeof(uint64_t))
        return -EINVAL;

    SETranslatingPortProxy proxy(tc);
    BufferArg maskBuf(cpu_set_mask, cpusetsize);
    maskBuf.copyIn(proxy);

    // マスクから最初にセットされているCPU番号を取得
    uint64_t mask = *(uint64_t *)maskBuf.bufferPtr();
    int target_cpu = -1;
    for (int i = 0; i < 64; i++) {
        if (mask & (1ULL << i)) {
            target_cpu = i;
            break;
        }
    }

    if (target_cpu < 0)
        return -EINVAL;

    // プロセスのaffinityマップに記録（キーはcpuIdを使用）
    auto process = tc->getProcessPtr();
    int cpu_key = tc->cpuId();
    process->setThreadAffinity(cpu_key, target_cpu);

    warn("sched_setaffinity: cpuId %d pinned to CPU %d (NUMA node %d)\n",
         cpu_key, target_cpu, process->getNumaNode(cpu_key));

    return 0;
#else
    warnUnsupportedOS("sched_setaffinity");
    return -1;
#endif
}
```

**意図：** `sched_setaffinity`の実装。affinityマスクからCPU番号を取り出し、
`Process::setThreadAffinity()`に記録する。

### 4. `src/arch/x86/linux/syscall_tbl64.cc`

**変更箇所：** syscall番号203の定義

```cpp
// 変更前

// 変更後

```

## 改造の効果

| 条件 | Node0 reads | Node1 reads |
|------|-------------|-------------|
| 改造前（全戦略） | 46162 | 0 |
| Scatter [0,8,1,9] | 30310 | 17309 |
| HPO [0,1,2,3] | 47103 | 0 |
| MPO [0,8,1,9] | 30300 | 17335 |

戦略によってNode0/Node1へのメモリアクセス比率が変化するようになった。

## 現在の限界と注意点

### cpuId=0問題

OpenMPはスレッドを`clone()`で生成するが、
新しいスレッドは全てCPU0のコンテキストから`sched_setaffinity`を呼ぶ。
そのため`tc->cpuId()`が全スレッドで0を返し、
affinityマップのキーが全て0で上書きされる。

```
# 実際のwarn出力
sched_setaffinity: cpuId 0 pinned to CPU 0 (NUMA node 0)
sched_setaffinity: cpuId 0 pinned to CPU 8 (NUMA node 1)  ← 全部cpuId=0
sched_setaffinity: cpuId 0 pinned to CPU 1 (NUMA node 0)
sched_setaffinity: cpuId 0 pinned to CPU 9 (NUMA node 1)
```

結果として`allocateMem`では最後に`setThreadAffinity`で記録されたノードが使われる。
4スレッドScatterの場合、最後の呼び出しがCPU9（Node1）なので
Node1へのメモリ割り当てが発生している。

### ノード構成変更時の対応

ノード数やコア数が変わる場合は`process.hh`の以下の行を変更する：

```cpp
int coresPerNode = 8;  // ← ここを変更
```

### first-touchポリシーの近似

本改造はLinuxカーネルの完全なfirst-touchポリシーを再現するものではなく、
`sched_setaffinity`の呼び出し順序に依存した近似実装である。
ヒープ領域の全てが正確にNUMAノードに配置される保証はない。

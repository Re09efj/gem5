# gem5 SE mode NUMA対応 改造ドキュメント（完成版）

## 背景

gem5のSE（Syscall Emulation）modeでNUMAシステムのスレッド配置戦略を研究するため、
以下を実現する必要があった。

1. OpenMPスレッドを指定したCPUに**実際に**配置する（`GOMP_CPU_AFFINITY`を機能させる）
2. 各スレッドが触るメモリを、そのスレッドが載るCPUの属するNUMAノードに配置する
3. 複数ノードにまたがってもOpenMPの同期（バリア）が正しく動く

最終的に、Scatter戦略[0,8]で以下の結果を得た（2スレッド）。

```
node0 (mem_ctrl0): reads=14390
node1 (mem_ctrl1): reads=8371
```

CPU0はNode0、CPU8はNode1のメモリを使い、戦略によって配分が変わる。

---

## 全体アーキテクチャ

4つの独立した仕組みを組み合わせることで実現する。

| # | 仕組み | 担当ファイル | 役割 |
|---|--------|-------------|------|
| 1 | スレッドのCPU強制配置 | `syscall_emul.hh`（doClone） | OpenMPワーカーを目的のCPUで起動 |
| 2 | 超軽量Dummy | `tiny_exit`（asm） | 非アクティブCPUを即Haltさせる |
| 3 | ページフォルト時のノード判定 | `mem_state.cc` / `process.*` / `faults.cc` | アクセス元CPUからNUMAノードを決定 |
| 4 | コヒーレントな単一fabric | `my_experiment.py` | ノード間でキャッシュ一貫性を保ちバリアを動かす |

このうち **#4が最重要**。ノード間でキャッシュコヒーレンシが無いと、
OpenMPバリアのカウンタ更新が他ノードに伝播せず、全スレッドが無限スピンして
デッドロックする。

// NUMA: thread affinity / clone制御
std::map<int, int> threadAffinityMap;
int coresPerNode = 8;       // Node0: CPU0-7, Node1: CPU8-15
int currentNumaNode = 0;    // 直近のページフォルトのノード
int cloneCount = 0;         // clone呼び出し回数（GEM5_TARGET_CPUSのインデックス）

void setThreadAffinity(int tid, int cpu_id) {
    threadAffinityMap[tid] = cpu_id;
    currentNumaNode = cpu_id / coresPerNode;
}
int getThreadAffinity(int tid) {
    auto it = threadAffinityMap.find(tid);
    return (it != threadAffinityMap.end()) ? it->second : 0;
}
int getNumaNode(int tid) {
    return getThreadAffinity(tid) / coresPerNode;
}

### 3. `src/sim/mem_state.hh` / `mem_state.cc`

`mem_state.hh` のシグネチャ変更。

```cpp
bool fixupFault(Addr vaddr, ThreadContext *tc = nullptr);
```

`mem_state.cc` に `#include "cpu/thread_context.hh"` を追加し、
`fixupFault` の冒頭でノードを決定。

```cpp
bool
MemState::fixupFault(Addr vaddr, ThreadContext *tc)
{
    // NUMA: ページフォルトを起こしたCPUのIDからノードを決定
    if (tc) {
        int cpu_id = tc->cpuId();
        // affinityマップが空のプロセス（Dummy）はNode0固定
        if (_ownerProcess->threadAffinityMap.empty()) {
            _ownerProcess->currentNumaNode = 0;
        } else {
            _ownerProcess->currentNumaNode =
                cpu_id / _ownerProcess->coresPerNode;
        }
    }
    // ... 以降は既存のVMA探索・allocateMem呼び出し ...
}
```

**意図：** SEモードでは物理ページ割り当てがページフォルト時の一点でのみ起きる。
このタイミングで「フォルトを起こしたCPU」が分かるので、そのCPUの属するノードに
ページを配置する。これが事実上の first-touch ポリシーになる。
Dummyプロセス（affinity未設定）はNode0に固定して余計なリモートアクセスを防ぐ。

#### (b) `doClone` ― OpenMPワーカーを目的CPUで起動

`findFree()` で空きThreadContextを取る箇所を、`GEM5_TARGET_CPUS`環境変数で
指定したCPUのThreadContextを選ぶように置き換える。

```cpp
ThreadContext *ctc;

// NUMA: GEM5_TARGET_CPUS から cloneCount番目のCPU番号を取得
auto *parent_process = tc->getProcessPtr();
int target_cpu_for_clone = -1;
for (const auto &env_str : parent_process->envp) {
    if (env_str.find("GEM5_TARGET_CPUS=") == 0) {
        std::string val = env_str.substr(17);
        int idx = 0;
        std::string token;
        std::istringstream ss(val);
        while (std::getline(ss, token, ',')) {
            if (idx == parent_process->cloneCount) {
                target_cpu_for_clone = std::stoi(token);
                break;
            }
            idx++;
        }
        break;
    }
}
parent_process->cloneCount++;

if (target_cpu_for_clone >= 0) {
    ctc = nullptr;
    for (auto *ctx : tc->getSystemPtr()->threads) {
        if (ctx && ctx->cpuId() == target_cpu_for_clone) {
            // Dummyが既にHaltedなら、そのtcをそのまま使う
            ctc = ctx;
            break;
        }
    }
    if (!ctc) ctc = tc->getSystemPtr()->threads.findFree();
} else {
    ctc = tc->getSystemPtr()->threads.findFree();
}

if (!ctc) {
    DPRINTF_SYSCALL(Verbose, "clone: no spare thread context in system"
                    "[cpu %d, thread %d]", tc->cpuId(), tc->threadId());
    return -EAGAIN;
}
```

**意図：** OpenMPのワーカースレッドが生成される `clone()` の時点で、
`active_indices[1:]` で指定したCPUのThreadContextを直接割り当てる。
これにより `gem5` 内部の `findFree()` 順序に依存せず、狙ったCPU
（例：CPU8=Node1）でワーカーが実際に動く。

既存の `archClone` がレジスタ（PC含む）を親tcから正しくコピーするため、
`doClone` 末尾は無改造でよい（`clearArchRegs` → `archClone` → `activate`）。

#### (c) `sched_setaffinity` ― affinityを記録するだけ

スレッド配置は `doClone` が済ませているので、ここでは**記録のみ**行う
（CPU移動やhaltContextは一切しない。O3CPUのLSQ assertionを避けるため）。

```cpp
template <class OS>
SyscallReturn
schedSetaffinityFunc(SyscallDesc *desc, ThreadContext *tc,
                     pid_t pid, typename OS::size_t cpusetsize,
                     VPtr<> cpu_set_mask)
{
#if defined(__linux__)
    if (cpusetsize < sizeof(uint64_t))
        return -EINVAL;

    SETranslatingPortProxy proxy(tc);
    BufferArg maskBuf(cpu_set_mask, cpusetsize);
    maskBuf.copyIn(proxy);

    uint64_t mask = *(uint64_t *)maskBuf.bufferPtr();
    int target_cpu = -1;
    for (int i = 0; i < 64; i++) {
        if (mask & (1ULL << i)) { target_cpu = i; break; }
    }
    if (target_cpu < 0)
        return -EINVAL;

    auto process = tc->getProcessPtr();
    process->setThreadAffinity(target_cpu, target_cpu);
    return 0;
#else
    warnUnsupportedOS("sched_setaffinity");
    return -1;
#endif
}
```

## 超軽量Dummyバイナリ `tiny_exit`

非アクティブなCPUにはDummyプロセスを割り当てるが、glibc付きバイナリだと
起動時にbrk等を呼び、`clone`が呼ばれる時点でまだHaltしていない。
これを避けるため、**syscall一発でexitするだけ**のバイナリを使う。

```asm
# tiny_exit.s
.globl _start
_start:
    mov $60, %rax    # sys_exit
    mov $0, %rdi     # exit code 0
    syscall
```

ビルド：

```bash
as -o /tmp/tiny_exit.o tiny_exit.s
ld -o ~/gem5/HomeGem5/binary/tiny_exit /tmp/tiny_exit.o
```

**意図：** glibcを介さず即座にexit→Haltするので、`doClone`が目的CPUの
ThreadContextを掴む時には確実にHalted状態になっている。

## gem5設定ファイル `my_experiment.py`（最重要）

### コヒーレントな単一fabric

**ノードごとにxbarを分けてBridgeで繋ぐ構成は使わない。**
`Bridge`はキャッシュコヒーレンシ（snoop）を伝播しないため、
ノードをまたいだOpenMPバリアがデッドロックする。

代わりに、**全CPUと全メモリコントローラを単一の `SystemXBar` に集約**する。
これによりMESIプロトコルが全CPUで機能し、バリアのカウンタ更新が
正しく伝播する。NUMAのノード区別は**物理アドレスのrange分割**で表現する。

```python
# 単一のコヒーレントfabric
system.membus = SystemXBar()

def connect_cpu_to_membus(cpu, membus):
    cpu.createInterruptController()
    cpu.interrupts[0].pio           = membus.mem_side_ports
    cpu.interrupts[0].int_requestor = membus.cpu_side_ports
    cpu.interrupts[0].int_responder = membus.mem_side_ports

    cpu.icache = L1ICache(); cpu.dcache = L1DCache()
    cpu.icache.cpu_side = cpu.icache_port
    cpu.dcache.cpu_side = cpu.dcache_port

    cpu.l2cache = L2Cache(); cpu.l2bus = L2XBar()
    cpu.icache.mem_side = cpu.l2bus.cpu_side_ports
    cpu.dcache.mem_side = cpu.l2bus.cpu_side_ports
    cpu.l2bus.mem_side_ports = cpu.l2cache.cpu_side
    cpu.l2cache.mem_side = membus.cpu_side_ports

    cpu.mmu.dtb.walker.port = membus.cpu_side_ports
    cpu.mmu.itb.walker.port = membus.cpu_side_ports

# 全CPUを接続
for cpu in all_cpus:
    connect_cpu_to_membus(cpu, system.membus)

# 各ノードのメモリをrange分割してmembusに接続（NUMAアドレスマップ）
for i in range(NUM_NODES):
    mc = MemCtrl()
    mc.dram = DDR3_1600_8x8(
        range=AddrRange(start=i * MEM_STRIDE, size=MEM_PER_NODE),
        ranks_per_channel=1,
    )
    mc.port = system.membus.mem_side_ports
    setattr(system, f"mem_ctrl{i}", mc)

system.system_port = system.membus.cpu_side_ports
```

### stats読み取り

単一membus構成ではメモリコントローラ名が `system.mem_ctrlN` になる。

```python
def read_mem_stats(stats_file, num_nodes):
    with open(stats_file) as f:
        content = f.read()
    for i in range(num_nodes):
        name = f"mem_ctrl{i}"
        reads  = re.search(rf"system\.{name}\.readReqs\s+(\d+)",  content)
        writes = re.search(rf"system\.{name}\.writeReqs\s+(\d+)", content)
        print(f"node{i} ({name}): "
              f"reads={reads.group(1) if reads else 0}, "
              f"writes={writes.group(1) if writes else 0}")
```

## 動作の流れ（まとめ）

```
1. CPU0でマスタースレッドが起動、初期化メモリをNode0に確保
2. OpenMPが clone() でワーカー生成
   → doClone が GEM5_TARGET_CPUS を読み、CPU8(Node1)のtcを割り当て
   → archClone がPC含むレジスタをコピー、CPU8で実行開始
3. sched_setaffinity がaffinityを記録（配置自体はdoClone済み）
4. CPU8が初めてメモリにアクセス → ページフォルト
   → fixupFault(tc) が cpuId=8 → Node1 と判定
   → allocateMem が Node1 の物理メモリからページ確保
5. バリア同期は単一membusのMESIで全CPU一貫 → 正常に完走
6. 結果: node0/node1 双方にreadsが分散
```

## 残課題

- **リモートレイテンシの非対称性**：現状は全CPUが単一membusに等距離で接続されており、
  ローカル/リモートのアクセスレイテンシ差がまだ無い。実機NUMAに近づけるには、
  各メモリへの到達経路にノード依存のレイテンシを付与する必要がある
  （コヒーレンシは維持したまま）。
- 全戦略（Scatter/MPO/HPO/Packed）× スレッド数（2/4/8/16）でのデータ収集。
